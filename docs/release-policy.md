# 发布边界说明

## 为什么不发布改包后的目标应用 APK

注入 BBZQ 后的目标应用 APK 仍然包含第三方闭源应用的原始代码、资源和应用主体。公开上传到 GitHub Release，本质上是在重新分发第三方闭源应用的修改版。

这和分享排障过程、patch 思路、构建脚本不同。后者分享的是方法，前者分发的是完整第三方软件副本。

因此本仓库不发布：

- 原始目标应用 APK。
- 注入模块后的完整目标应用 APK。
- 签名文件、证书、密钥。
- 含设备标识的完整日志。

## 可以发布什么

可以发布：

- 技术复盘。
- 构建脚本。
- NPatch 兼容补丁。
- 日志判断点。
- BBZQ APK 的校验信息。
- 在许可证允许前提下发布的开源项目构建产物。

BBZQ 本身是开源项目，能否发布其 APK 取决于其许可证和对应源代码提供方式。当前记录中，BBZQ 使用木兰公共许可证第 2 版，允许在满足许可证要求的前提下分发可执行形式。

## 为什么应用商店能保留旧版 APK

应用商店和个人 GitHub 仓库的风险模型不同。

常见情况包括：

- 应用商店可能与应用开发者或版权方有分发合作、开发者后台上传、授权或历史渠道包来源。
- 应用商店通常有专门的开发者协议、内容审核、下架处理、投诉处理和合规团队。
- 应用商店分发的一般是原始官方包或渠道包，不一定是个人重新签名、修改后的完整 APK。
- 即使某个平台能下载旧版 APK，也不等于任何个人都自动获得再分发该 APK 或其修改版的权利。

关键区别：

| 场景 | 分发对象 | 风险 |
| --- | --- | --- |
| 应用商店旧版下载 | 可能是官方包、渠道包或经平台处理的应用分发 | 平台承担授权、审核、投诉处理等责任 |
| 个人 GitHub Release 发布改包 | 第三方闭源应用的修改版完整 APK | 容易构成未授权再分发 |
| 本仓库分享技术细节 | 文档、脚本、补丁、日志判断点 | 风险较低，且更符合经验分享目的 |

所以，本仓库采用“公开技术细节 + 本地自行构建”的方式。

这种方式仍能减少后来者踩坑，但不会把第三方闭源应用完整副本放到公开 Release。
